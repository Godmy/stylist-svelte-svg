# CODEX-002-FORKS-AND-NPM-DISTRIBUTION

## Дата
2026-05-01

## Тема
Forks для развития библиотеки и npm как рыночная форма поставки

## Короткий ответ
Да, в общем виде ты мыслишь правильно, но формулировку нужно сделать точнее:

- `forks` не являются основным способом "распространения" библиотеки клиентам
- `forks` являются одним из способов организации работы над исходниками
- для рынка библиотека обычно поставляется как `npm` package

То есть:

1. для разработки и совершенствования ядра возможны `fork + pull request`
2. для потребления на рынке почти всегда нужна поставка через `npm`

## Где ты прав
Ты правильно разделяешь две реальности:

1. **развитие исходного кода**
2. **поставка готового библиотечного продукта**

Это действительно две разные модели.

## Что лучше считать основной схемой
Для твоего случая я бы сформулировал так:

### 1. Upstream-модель разработки
Главный репозиторий `stylist-svelte` живет как upstream.

Дальше:
- ты работаешь напрямую в upstream
- внешние участники могут работать через `fork`
- изменения возвращаются через `pull request`

Это стандартная open-source и team-collaboration модель.

### 2. Рыночная модель распространения
Потребители библиотеки не должны брать ее через `fork`, если они не собираются менять саму библиотеку.

Для рынка нормальная форма:
- `npm package`
- документация
- semver-версии
- changelog
- примеры интеграции

То есть покупатель курса или клиент обычно должен:

1. установить пакет
2. прочитать документацию
3. использовать публичный API

а не форкать исходный репозиторий как основной сценарий.

## Важное различие
### Fork нужен тем, кто хочет менять библиотеку
Это сценарий:
- контрибьютор
- партнер-разработчик
- internal team
- advanced student, который лезет в ядро

### npm нужен тем, кто хочет пользоваться библиотекой
Это сценарий:
- клиент
- ученик курса
- команда продукта
- downstream-проект

## Для обучающего курса
Тут лучше думать в двух слоях.

### Слой 1. Курс как обучение использованию
Если курс про то, как применять библиотеку, то студентам нужен:
- `npm` пакет
- документация
- демо-проект
- возможно starter template

### Слой 2. Курс как обучение развитию самой библиотеки
Если курс про архитектуру и эволюцию `stylist-svelte`, тогда уже нужен:
- доступ к upstream repo
- возможно forks
- упражнения через PR
- разбор contribution flow

То есть forks для курса нужны не всем студентам, а только в той части курса, где они учатся менять саму библиотеку.

## Для коммерческой модели
Если смотреть прагматично, рынок обычно лучше понимает такую схему:

1. есть библиотека как продукт
2. она ставится из `npm`
3. есть документация и примеры
4. есть курс, который учит с ней работать
5. есть отдельный уровень для контрибьюторов или enterprise-партнеров

Это намного понятнее, чем схема "покупатель форкает репозиторий и так потребляет библиотеку".

## Где forks реально полезны
Forks полезны как:

- contribution model
- experimental branches на уровне отдельных репозиториев
- white-label / research / deep customization
- учебный материал для advanced track

Но forks не стоит делать основным каналом поставки библиотеки пользователям.

## Более точная формула
Не "распространять репозиторий через форки, а на рынке сборка на npm", а так:

**Развитие библиотеки:**
- upstream repo
- branches
- forks при необходимости внешнего contribution

**Распространение библиотеки:**
- npm package
- docs
- examples
- starter repos

## Что я бы рекомендовал лично тебе
Для `stylist-svelte` я бы строил модель в таком порядке:

1. Главный repo библиотеки как source of truth.
2. `npm` как основной канал потребления библиотеки.
3. `forks` как механизм collaboration, а не продаж.
4. Для курса:
   - базовый поток через установку пакета
   - продвинутый поток через работу с исходниками и PR/fork model

## Если совсем коротко
Да:

- для совершенствования библиотеки `forks` могут быть частью процесса
- для рынка библиотеку лучше поставлять через `npm`

Но:

- не надо делать `fork` главным способом доставки библиотеки клиенту
- `fork` это инструмент разработки
- `npm` это инструмент распространения

## Практический вывод
Если ты хочешь выглядеть как зрелый библиотечный проект, то внешняя витрина должна быть package-oriented, а не fork-oriented.

То есть снаружи:
- `npm`
- docs
- examples

А внутри процесса разработки:
- upstream
- branches
- forks
- PR
