# CODEX-001-GITHUB-FORKS

## Дата
2026-05-01

## Тема
Forks, submodule, subtree и модель развития `stylist-svelte` как библиотеки

## Короткий ответ
Да, сделать это можно, но тут важно разделить две разные задачи:

1. `fork` в GitHub решает задачу ветвления владения и contribution flow.
2. `submodule` / `subtree` / package dependency решают задачу встраивания библиотеки в другой репозиторий.

То есть fork сам по себе не является хорошей заменой механизму подключения библиотеки в продукт.

## Что именно ты, по сути, хочешь
Судя по описанию, ты хочешь одновременно получить:

1. Отдельную жизнь библиотеки `stylist-svelte`.
2. Возможность создавать новые продуктовые репозитории поверх нее.
3. Возможность править библиотеку из продуктового репозитория.
4. Возможность коммитить изменения и в библиотечный репозиторий, и в продуктовый репозиторий.

Это нормальная цель. Но для нее есть несколько принципиально разных моделей.

## Важное ограничение
Один и тот же каталог с файлами не может "просто так" быть одновременно обычной частью двух независимых git-репозиториев.

Если хочется, чтобы `stylist-svelte/src/lib` жило как самостоятельный git-источник внутри другого репозитория, стандартный путь здесь:

- либо `git submodule`
- либо `git subtree`
- либо отдельный package/repo и подключение через package manager
- либо монорепа/workspace

## Вариант 1. Отдельный repo библиотеки + `git submodule`
### Как работает
В продуктовом репозитории у тебя лежит папка `stylist-svelte/`, но это не обычная папка, а вложенный git-репозиторий.

### Что это дает
- Можно зайти в `stylist-svelte` и коммитить прямо в библиотеку.
- Внешний репозиторий коммитит только ссылку на конкретный commit submodule.
- У библиотеки остается своя история.
- Продукт может фиксироваться на нужной версии библиотеки.

### Что важно
- Это реально позволяет "коммитить и туда, и туда", но это будут два разных коммита:
  - один в repo библиотеки
  - второй в repo продукта, который обновляет pointer на новый commit submodule

### Плюсы
- Чистое разделение ответственности.
- Нормально подходит для library-first архитектуры.
- Хорошо работает, если `stylist-svelte` должна жить независимо.

### Минусы
- Submodule требует дисциплины.
- Люди часто забывают обновить pointer в родительском repo.
- Для команды это чуть сложнее, чем обычная папка.

## Вариант 2. Отдельный repo библиотеки + `git subtree`
### Как работает
Код библиотеки физически копируется в продуктовый repo, но с сохранением связи истории через subtree-механизм.

### Что это дает
- В продуктовом repo библиотека выглядит как обычная папка.
- Проще для разработчиков, чем submodule.
- Можно подтягивать изменения из library repo и отправлять их обратно.

### Плюсы
- Удобнее daily workflow.
- Нет отдельного шага инициализации submodule.
- Внешне проще для CI и для новых участников.

### Минусы
- История и синхронизация сложнее концептуально.
- Легко превратить библиотеку в "случайно форкнутую копию", если не держать дисциплину.
- Обратная отправка изменений в library repo менее прозрачна, чем в submodule.

## Вариант 3. Отдельный repo библиотеки + пакетная зависимость
### Как работает
`stylist-svelte` живет как npm package, а продукт подключает ее как зависимость.

### Что это дает
- Самая чистая библиотечная модель.
- Продукт зависит от версий, а не от исходников в дереве.
- Хорошо масштабируется на много downstream-репозиториев.

### Плюсы
- Наиболее зрелая модель для библиотеки.
- Простое versioning, release, changelog.
- Хорошо подходит, если downstream не должен править ядро напрямую.

### Минусы
- Менее удобно, если ты хочешь разрабатывать библиотеку и продукт одновременно в одном рабочем дереве.
- Для локальной совместной разработки нужны `workspace`, `link`, `pnpm/yarn workspaces` или локальный registry flow.

## Вариант 4. Монорепа
### Как работает
`stylist-svelte` и продукт лежат в одном общем repo как пакеты workspace.

### Что это дает
- Самый удобный DX для одновременной разработки.
- Изменения библиотеки и продукта можно коммитить атомарно одним commit.
- Нет боли submodule.

### Плюсы
- Лучший вариант, если библиотека пока развивается вместе с продуктами.
- Очень удобен для refactor и массовых архитектурных изменений.

### Минусы
- Библиотека теряет полную независимость на уровне repo.
- Сложнее отделять публичный продукт библиотеки от внутренних потребителей.
- Если твоя стратегическая цель именно "много независимых репозиториев", это уже не совсем то.

## Вариант 5. GitHub fork как модель downstream
### Как работает
Каждый новый продукт стартует как fork репозитория библиотеки или как fork базового шаблона.

### Когда это уместно
- Когда downstream должен сильно расходиться с upstream.
- Когда это скорее независимая производная система, а не просто потребитель библиотеки.

### Проблема
Fork удобен для старта и contribution, но плох как долгосрочный механизм переиспользования общей библиотеки.

Почему:
- быстро появляется divergence
- тяжело регулярно подтягивать upstream
- продукт и библиотека начинают смешиваться
- архитектурно это уже не "библиотека + потребители", а "семейство родственных репозиториев"

## Ответ на твой прямой вопрос
### "Можно ли сделать так, чтобы были доступны файлы `stylist-svelte/src/lib`, чтобы можно было коммитить в `stylist-svelte` и коммитить в другой репозиторий?"
Да, но корректно это решается так:

- если нужен отдельный git-источник внутри другого repo: `submodule`
- если нужна обычная папка с возможностью sync: `subtree`
- если нужен зрелый library workflow: отдельный package + dependency
- если нужен лучший DX для совместной разработки: монорепа/workspace

Самая буквальная реализация твоего сценария - это именно `submodule`.

## Практическая рекомендация для твоего случая
Если `stylist-svelte` для тебя именно библиотека, я бы разделил стратегию на два уровня:

### Базовая стратегия
1. Сделать `stylist-svelte` отдельным главным repo библиотеки.
2. Продуктовые репозитории создавать отдельно.
3. Подключать библиотеку в продукты либо через `submodule`, либо через package dependency.

### Что выбрать
- `submodule`, если тебе нужен живой доступ к исходникам библиотеки внутри продуктового repo и частые правки "на месте".
- package dependency, если ты хочешь более зрелую модель release/versioning.
- монорепа, если проект пока еще в активной фазе архитектурной сборки и разделять repo рано.

## Мой инженерный вывод
Если цель: "я строю платформенную библиотеку и хочу на ее базе делать разные репозитории", то я бы не ставил `fork` в центр стратегии.

Я бы ставил в центр:

1. отдельный repo библиотеки
2. четкую модель подключения библиотеки
3. понятный upstream/downstream flow

Forks здесь полезны как вспомогательный механизм, но не как основная архитектура развития.

## Рекомендация по порядку зрелости
Если идти без лишней сложности:

1. Сначала решить, нужен ли тебе независимый repo для `stylist-svelte`.
2. Если да, начать с `submodule`, если важна совместная разработка через исходники.
3. Когда API библиотеки стабилизируется, перейти к package-based модели.

## Следующий полезный вопрос
Следующая точка, которую реально стоит разобрать: что именно у тебя является "продуктом" относительно `stylist-svelte`:

- сайт
- приложение
- шаблон
- white-label fork
- demo/playground
- отдельная дизайн-система клиента

От этого уже зависит, нужен тебе `submodule`, `subtree`, monorepo или package registry.
