# План подготовки `stylist-svelte` к релизу

Дата: 2026-05-06

## Текущее состояние

- Опубликованная версия на npm: `1.0.0`
- `dist.unpackedSize` на npm: `4 951 112` байт
- Количество файлов в опубликованном пакете: `4038`
- Локальная версия в репозитории: `1.0.1`
- Важное изменение: `lucide-svelte` был удален из проекта, потому что набора иконок не хватало для нужд библиотеки

## Главная цель следующего релиза

Подготовить следующий выпуск библиотеки так, чтобы он был:

- воспроизводимым по сборке
- предсказуемым по публичному API
- аккуратным по составу npm-пакета
- понятным для потребителя по документации
- устойчивым по стратегии иконок после отказа от `lucide-svelte`

## Основные проблемы, которые нужно закрыть

### 1. Несоответствие версии между npm и репозиторием

Сейчас в npm опубликована `1.0.0`, а в репозитории уже указана `1.0.1`.

Что нужно сделать:

- решить, какой номер будет у следующего релиза
- зафиксировать, что именно вошло после `1.0.0`
- синхронизировать `package.json`, changelog и релизный тег

Критерий завершения:

- есть согласованная целевая версия следующего релиза

### 2. Слишком большой и не до конца контролируемый состав пакета

Размер опубликованного пакета уже заметный, а число файлов очень большое для UI-библиотеки. Нужно понять, что именно реально требуется потребителю.

Что нужно сделать:

- прогнать `npm pack --dry-run --json`
- разобрать, какие файлы из `dist` действительно нужны пользователю библиотеки
- исключить из публикации `index.story.svelte`, demo-файлы, внутренние артефакты анализа и все, что не относится к runtime/public API
- проверить `files` в `package.json`
- отдельно проверить, не публикуются ли лишние JSON/SVG-артефакты, нужные только внутренним инструментам

Критерий завершения:

- состав tarball полностью осознан и контролируем
- размер пакета уменьшен настолько, насколько это возможно без поломки API

### 3. Надежность сборки и обязательных проверок

Релиз нельзя выпускать, если quality gates проходят нестабильно или только вручную.

Что нужно сделать:

- добиться стабильного прохождения `yarn build`
- добиться стабильного прохождения `yarn validate`
- после изменений, влияющих на экспорты, обязательно прогонять:
  - `python -u "d:\2026\code\template\stylist\indexation\cli.py"`
- перед релизом обязательно прогонять:
  - `python -u "d:\2026\code\template\stylist\errors\cli.py"`
- проверить, что `prepublishOnly` не опирается на устаревшие локальные артефакты

Критерий завершения:

- release candidate проходит build, lint, typecheck, svelte-check, tests, indexation и unified error analysis

### 4. Фиксация публичного API

Нужно четко определить, что библиотека обещает пользователю как поддерживаемый контракт.

Что нужно сделать:

- определить список публичных экспортов
- убрать случайные или внутренние экспорты, которые попадают наружу только из-за генерации
- решить, является ли поддерживаемым контрактом только root export `"."`, или subpath imports тоже считаются частью публичного API
- проверить generated barrels на случайные утечки внутренних сущностей

Критерий завершения:

- публичный API зафиксирован и документирован

### 5. Подготовка документации для потребителя

Документация релиза должна объяснять, как реально пользоваться библиотекой, а не только описывать ее идеологию.

Что нужно сделать:

- усилить `README.md`: установка, peer dependencies, быстрый старт, требования к темизации, базовые сценарии использования
- добавить `CHANGELOG.md`
- исправить битые ссылки в документации
- если `CONTRIBUTING.md` ссылается на `CODE_OF_CONDUCT.md`, то либо добавить этот файл, либо убрать ссылку
- описать новую стратегию иконок после удаления `lucide-svelte`
- добавить migration notes, если следующий релиз меняет структуру пакета или способ работы с иконками

Критерий завершения:

- новый пользователь может поставить пакет и начать использовать библиотеку, опираясь только на README

### 6. Новая стратегия иконок после удаления `lucide-svelte`

Удаление `lucide-svelte` решает одну проблему, но создает другую: теперь нужно явно определить источник и жизненный цикл иконок.

Что нужно сделать:

- определить канонический источник иконок для библиотеки
- задокументировать, как иконки хранятся, генерируются и обновляются
- проверить, что в коде и документации не осталось скрытых зависимостей от `lucide-svelte`
- проверить, не публикуется ли лишний объем иконок, который не нужен в runtime
- убедиться, что библиотека не окажется снова в ситуации ручного аварийного добора иконок

Критерий завершения:

- стратегия иконок явно описана, устойчива и не зависит от старых предположений про `lucide-svelte`

## Рекомендуемая последовательность работ

### Этап 1. Зафиксировать релизную цель

1. Определить целевую версию после `1.0.0`.
2. Зафиксировать состав изменений, которые войдут в релиз.
3. Подготовить или обновить `CHANGELOG.md`.

### Этап 2. Очистить публикуемый артефакт

1. Прогнать `npm pack --dry-run --json`.
2. Удалить из публикуемого результата development-only файлы.
3. Повторно прогнать dry-run и сравнить размер и количество файлов.

### Этап 3. Довести обязательные проверки до зеленого состояния

1. Прогнать indexation.
2. Прогнать unified error analysis.
3. Прогнать локальную валидацию и тесты.
4. Исправить все проблемы, пока release gate не станет полностью зеленым.

### Этап 4. Зафиксировать API и документацию

1. Проверить экспортируемую поверхность.
2. Ужать и переписать README под потребителя.
3. Задокументировать иконную архитектуру и ожидания от нее.

### Этап 5. Финальная проверка перед публикацией

1. Подтвердить версию в `package.json`.
2. Подтвердить финальный состав tarball через `npm pack --dry-run`.
3. Подтвердить зеленый CI на релизном коммите.
4. Выполнить `npm publish`.
5. Проверить результат на странице npm: версию, размер, file count.

## Краткий релизный checklist

- согласована целевая версия
- подготовлен changelog
- обновлен README для потребителей
- нет битых ссылок в документации
- описана стратегия иконок
- проверен публичный API
- проверен состав tarball
- stories и demo исключены из публикации, если они не часть публичного контракта
- `yarn build` проходит
- `yarn validate` проходит
- после экспортных изменений прогнан `python -u "d:\2026\code\template\stylist\indexation\cli.py"`
- перед релизом проходит `python -u "d:\2026\code\template\stylist\errors\cli.py"`
- CI зеленый на релизном коммите
- публикация в npm выполнена
- post-publish проверка завершена

## Рекомендуемые ближайшие действия

1. Сначала выровнять версионирование: решить, следующий релиз это `1.0.1` или уже другая версия.
2. Затем разобрать содержимое npm-пакета и убрать из `dist` все лишнее.
3. После этого добавить `CHANGELOG.md`.
4. Обновить README и отдельно описать стратегию иконок после удаления `lucide-svelte`.
5. Только потом запускать полный release gate и готовить publish.
