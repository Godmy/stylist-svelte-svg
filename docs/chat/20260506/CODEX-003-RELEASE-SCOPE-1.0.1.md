# Scope ближайшего релиза `stylist-svelte`

Дата: 2026-05-06

## Предпосылка

Сейчас на npm опубликована версия `1.0.0`, а в репозитории локально уже указана `1.0.1`.

По текущему состоянию проекта ближайший релиз разумно трактовать как стабилизирующий и гигиенический, а не как большой функциональный выпуск. Это значит, что в `1.0.1` лучше включать только то, что:

- повышает надежность публикации
- уменьшает размер и шум npm-пакета
- делает API и документацию понятнее
- завершает переход после удаления `lucide-svelte`

## Рекомендуемый смысл версии `1.0.1`

`1.0.1` стоит выпускать как:

- patch-релиз по semver
- релиз стабилизации publish pipeline
- релиз очистки пакета
- релиз фиксации документации и иконной стратегии

Не стоит превращать `1.0.1` в релиз новых крупных компонентных семейств или архитектурных экспериментов.

## Что включить в `1.0.1`

### 1. Выравнивание версии и релизной метаинформации

Включить:

- финальную синхронизацию версии `1.0.1`
- `CHANGELOG.md` для различий между `1.0.0` и `1.0.1`
- release notes с коротким объяснением сути релиза

Почему это должно войти:

- без этого релиз невозможно нормально объяснить пользователю

### 2. Очистку npm-пакета

Включить:

- исключение story/demo/development-only файлов из publish-артефакта, если они не являются частью публичного контракта
- пересмотр состава `dist`
- пересмотр `files` в `package.json`
- повторную проверку через `npm pack --dry-run --json`

Почему это должно войти:

- опубликованный пакет уже тяжелый
- file count слишком большой
- это прямое улучшение потребительского качества пакета

### 3. Стабилизацию release gates

Включить:

- исправления, необходимые для стабильного прохождения `yarn build`
- исправления, необходимые для прохождения `yarn validate`
- обязательный прогон indexation и unified error analysis перед релизом

Почему это должно войти:

- публикация без надежных quality gates создаст технический долг уже после `1.0.0`

### 4. Документацию для потребителя

Включить:

- обновление `README.md`
- добавление или исправление `CHANGELOG.md`
- устранение битых ссылок в документации
- согласование `CONTRIBUTING.md` с фактической структурой проекта

Почему это должно войти:

- это low-risk улучшения с высоким эффектом для восприятия библиотеки

### 5. Фиксацию новой стратегии иконок

Включить:

- документирование замены `lucide-svelte`
- проверку остатков старой зависимости в коде, скриптах и документации
- фиксацию текущего канонического источника иконок

Почему это должно войти:

- удаление `lucide-svelte` уже изменило внутреннюю модель библиотеки
- это надо оформить как осознанное решение, а не как временный обход

## Что можно включить только если это не раздувает релиз

### 1. Небольшие исправления экспортов

Можно включить:

- локальные исправления barrel exports
- мелкие корректировки public API, если они не ломают пользователей

Условие:

- изменения не должны тянуть за собой массовую архитектурную миграцию

### 2. Локальные исправления сборки

Можно включить:

- правки build-конфига
- правки publish-конфига
- правки исключений для лишних файлов

Условие:

- они должны быть прямо связаны с релизом `1.0.1`

### 3. Мелкие исправления README и onboarding

Можно включить:

- минимальный пример установки
- описание peer dependencies
- краткое описание theming/runtime ожиданий

Условие:

- не превращать это в полную документационную реформу

## Что не стоит включать в `1.0.1`

### 1. Крупные новые компоненты и домены

Не включать:

- большие новые семейства компонентов
- новые домены, если они не нужны для release hygiene
- масштабные UI-фичи

Почему:

- это увеличивает риск регрессий и размывает смысл patch-релиза

### 2. Архитектурные реформы широкого масштаба

Не включать:

- новые архитектурные соглашения
- большие миграции структуры `src/lib`
- переизобретение barrel/indexation механики сверх необходимого для релиза

Почему:

- такие изменения лучше выносить в `minor` или отдельную стабилизационную ветку

### 3. Массовую косметическую переработку всего репозитория

Не включать:

- тотальную перепаковку документации
- глобальный форматинг всего проекта, если он не нужен для прохождения release gates
- широкие рефакторы без прямой пользы для publish

Почему:

- это повышает шум diff и усложняет релизную проверку

## Что стоит отложить на следующий `minor`

Перенести в следующий `minor`:

- расширение публичного API
- новые крупные наборы иконок или новый большой icon tooling
- большие изменения структуры экспортов
- новые функциональные блоки библиотеки
- системную переработку документационного портала

## Блокеры выпуска `1.0.1`

Следующие пункты должны считаться блокирующими:

- `yarn build` не проходит стабильно
- `yarn validate` не проходит
- итоговый npm tarball содержит явно лишние story/demo/dev-only файлы без осознанного контракта
- нет changelog для различий между `1.0.0` и `1.0.1`
- README не объясняет базовый сценарий использования пакета
- не определено, как библиотека теперь живет без `lucide-svelte`

## Практический рекомендуемый scope `1.0.1`

Включить обязательно:

- release metadata
- changelog
- README cleanup
- package cleanup
- publish artifact cleanup
- build and validation stabilization
- icon strategy documentation

Включить опционально:

- точечные безопасные исправления экспортов
- точечные исправления release tooling

Не включать:

- новые большие features
- крупные архитектурные миграции
- большие функциональные расширения библиотеки

## Рекомендуемая формулировка релиза

`1.0.1` — это релиз стабилизации публикации и упаковки библиотеки после `1.0.0`, с очисткой npm-пакета, уточнением документации и фиксацией стратегии иконок после отказа от `lucide-svelte`.

## Следующий шаг после `1.0.1`

После выпуска `1.0.1` уже можно готовить следующий `minor`, в котором отдельно планировать:

- расширение возможностей библиотеки
- эволюцию публичного API
- новые компонентные семейства
- более глубокую документационную систему
