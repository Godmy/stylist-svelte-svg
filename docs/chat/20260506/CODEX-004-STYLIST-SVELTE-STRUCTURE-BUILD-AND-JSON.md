# Карта папок `stylist-svelte`: библиотека, сайт, билд и JSON

Дата: 2026-05-06

## 1. Что находится в `stylist-svelte` по папкам

Ниже разложено, что делает каждая верхнеуровневая папка, что попадает в npm-библиотеку, и что участвует в работе сайта/сервера на SvelteKit.

## 2. Верхний уровень проекта

### `.claude`

- Назначение: служебная папка для локальной агентной/IDE-интеграции
- В npm-библиотеку: не попадает
- В сайт/серверный билд: не попадает

### `.github`

- Назначение: CI и workflow проекта
- В npm-библиотеку: не попадает
- В сайт/серверный билд: не попадает
- Практический смысл: влияет на контроль качества релиза, но не на runtime пакета

### `.logs`

- Назначение: локальные логи разработки
- В npm-библиотеку: не попадает
- В сайт/серверный билд: не должно попадать

### `.svelte-kit`

- Назначение: служебный build cache и промежуточные артефакты SvelteKit
- В npm-библиотеку: не попадает
- В сайт/серверный билд: это внутренняя build-папка dev/build процесса, но не источник проекта

### `.yarn`

- Назначение: служебные артефакты Yarn
- В npm-библиотеку: не попадает
- В сайт/серверный билд: не попадает

### `adr`

- Назначение: архитектурные решения проекта
- В npm-библиотеку: не попадает
- В сайт/серверный билд: не попадает
- Практический смысл: влияет на правила разработки, но не на артефакт для потребителя

### `dist`

- Назначение: собранный output библиотеки после `svelte-package`
- В npm-библиотеку: попадает целиком через `files` в `package.json`
- В сайт/серверный билд: обычно не является источником для dev-сайта; это publish artifact библиотеки
- Важный факт: сейчас именно `dist` является основным содержимым npm-пакета

### `docs`

- Назначение: документация, заметки, внутренние материалы
- В npm-библиотеку: не попадает напрямую
- В сайт/серверный билд: не попадает как runtime-часть сайта автоматически
- Исключение: если что-то из docs вручную импортируется в код, тогда влияет, но по текущей структуре это выглядит как вне-runtime слой

### `node_modules`

- Назначение: зависимости
- В npm-библиотеку: не попадает
- В сайт/серверный билд: используется только как зависимостная база при сборке

### `scripts`

- Назначение: генераторы, скрипты подготовки данных, JSON-артефактов и технических слоев
- В npm-библиотеку: не попадает напрямую
- В сайт/серверный билд: не является runtime-слоем сайта
- Практический смысл: это build-time/tooling слой проекта

### `src`

- Назначение: главный исходный код проекта
- В npm-библиотеку: попадает косвенно, через упаковку в `dist`
- В сайт/серверный билд: это основной исходник SvelteKit-приложения
- Важно: здесь одновременно живут и библиотека, и демо/сайт/маршруты

### `static`

- Назначение: статические ассеты сайта
- В npm-библиотеку: не попадает через `files`
- В сайт/серверный билд: попадает как публичные статические файлы SvelteKit
- По факту сейчас здесь лежат:
  - `favicon.png`
  - `logos/*`

## 3. Что находится в `src`

В `src` есть три основных слоя:

- `src/lib`
- `src/routes`
- `src/test`

### `src/lib`

- Назначение: исходники самой библиотеки
- В npm-библиотеку: да, через сборку в `dist`
- В сайт/серверный билд: да, потому что SvelteKit-сайт использует эти же исходники напрямую

### `src/routes`

- Назначение: страницы, серверные endpoint'ы и демо-слой сайта
- В npm-библиотеку: напрямую не должны попадать как часть публичной библиотеки
- В сайт/серверный билд: да, это главный слой сайта/приложения

Текущие маршруты и их смысл:

- `+layout.svelte`
  - общий layout сайта
- `+page.svelte`
  - корневая страница сайта
- `+page.ts`
  - клиентская/страничная логика корневого маршрута
- `app.svelte`
  - прикладной корневой Svelte-файл
- `api/error-log/+server.ts`
  - серверный endpoint
- `domain/+page.server.ts`
  - серверная логика страницы доменного просмотра
- `domain/+page.svelte`
  - UI страницы доменного просмотра
- `domain/content/+server.ts`
  - серверный endpoint для контента
- `stylist/+page.svelte`
  - страница, связанная с библиотекой/демо
- `stylist/stylist-demo-page.svelte`
  - демонстрационная страница
- `_lib/server/error-log.ts`
  - серверный helper для роутов

### `src/test`

- Назначение: тестовый слой
- В npm-библиотеку: не должен попадать
- В сайт/серверный билд: не является runtime-слоем
- По факту сейчас присутствует дерево `theme/test/runtime`

## 5. Как читать эти домены

Каждый такой домен в `src/lib` является частью библиотеки как исходного кода. При упаковке он, как правило, отражается в `dist` почти симметрично.

Это значит:

- все домены из `src/lib`, если они экспортируются или входят в `dist`, фактически становятся частью npm-артефакта
- те же домены одновременно доступны сайту в dev/build режиме, если страницы из `src/routes` их импортируют

Иначе говоря:

- `src/lib` = исходная библиотека
- `dist` = упакованная библиотека
- `src/routes` = сайт/демо/серверные страницы поверх библиотеки

## 6. Что именно попадает в npm-библиотеку

По `package.json` в npm-пакет попадает:

- `dist`
- при этом исключаются только `dist/**/*.test.*` и `dist/**/*.spec.*`

То есть в библиотеку сейчас попадает не только runtime-код компонентов и типов, но вообще весь собранный `dist`, если он оказался там после `svelte-package`.

Практически это означает:

- все собранные домены из `src/lib` попадают в библиотеку
- если в `dist` попали story/demo/dev-only файлы, они тоже попадают в npm-пакет
- проблема состава пакета решается не на уровне `src/routes`, а в первую очередь на уровне того, что оказывается в `dist`

## 7. Что попадает в билд сайта / для доступа на сервере

Если смотреть на проект как на SvelteKit-приложение, то в сайт/серверный слой входят:

- `src/routes`
- `src/lib`
- `static`
- часть конфигурации из `svelte.config.js`, `vite.config.ts`, `tsconfig.json`
- служебный output `.svelte-kit` в процессе сборки

Что это значит по ролям:

- `src/routes` определяет страницы и endpoint'ы
- `src/lib` дает страницам компоненты, логику и данные
- `static` дает публичные файлы
- `api/*/+server.ts` и другие `+server.ts` дают серверные точки входа

Важная развилка:

- npm-пакет библиотеки живет через `dist`
- SvelteKit-сайт живет через `src/routes` + `src/lib` + `static`

Это два разных артефакта, хотя исходники у них частично общие.

## 9. Отдельно: что я знаю точно

Ниже только факты, подтвержденные по текущему проекту.

### Факт 1. npm-публикация идет из `dist`

Это подтверждается `package.json`:

- `exports["."].types = "./dist/index.d.ts"`
- `exports["."].svelte = "./dist/index.js"`
- `files = ["dist", "!dist/**/*.test.*", "!dist/**/*.spec.*"]`

Следствие:

- все, что попало в `dist`, почти наверняка идет в npm-пакет

### Факт 2. сайт и библиотека живут в одном проекте

Это подтверждается структурой:

- библиотека находится в `src/lib`
- сайт находится в `src/routes`

Следствие:

- проект совмещает package-repo и demo/site-repo в одном дереве

### Факт 3. у сайта есть серверные маршруты

Это видно по:

- `src/routes/api/error-log/+server.ts`
- `src/routes/domain/+page.server.ts`
- `src/routes/domain/content/+server.ts`

Следствие:

- это не просто статическая витрина компонентов, а SvelteKit-приложение с серверной логикой

### Факт 4. `src/lib` и `dist` зеркалят друг друга по доменам

Список верхнеуровневых доменов в `src/lib` и `dist` совпадает по именам.

Следствие:

- библиотечные домены упаковываются почти целиком

### Факт 5. JSON-слой очень большой и распределенный

Подтверждено:

- в `scripts/json` сейчас `49` файлов
- в `src/lib` сейчас `274` JSON-файла

Следствие:

- JSON в этом проекте не вспомогательная мелочь, а заметная часть архитектуры

### Факт 6. `static` сейчас относится к сайту, а не к npm-пакету

По текущей схеме публикации `static` не входит в `files` пакета.

Следствие:

- изображения и логотипы из `static` работают для сайта, но не публикуются как часть npm-библиотеки автоматически

### Факт 7. `scripts` это build-time слой, а не runtime слой

Скрипты запускаются через `tsx` и npm scripts:

- `data:*`
- `const:*`
- `type:*`
- `interface:*`
- `class:*`
- `function:*`
- `json:meta`

Следствие:

- `scripts` служит для генерации и анализа, а не для прямого использования потребителем библиотеки

## 10. Отдельно: что касается JSON-файлов в проекте

JSON здесь нужно разделять минимум на три разных класса.

## 11. Класс 1. JSON в `scripts/json`

Это build-time и analytical JSON-слой.

Основные подпапки:

- `scripts/json/build`
- `scripts/json/catalogs`
- `scripts/json/core`
- `scripts/json/data`
- `scripts/json/dependency`
- `scripts/json/meta`
- `scripts/json/reports`
- `scripts/json/samo`

Что это означает:

- это внутренние каталоги, отчеты, агрегаты, метаданные и служебные результаты
- они нужны для генерации, анализа, inventory, SAMO-слоя и внутренних процессов
- это не обязательный публичный runtime API библиотеки

С практической точки зрения:

- эти JSON не должны публиковаться в npm-пакет автоматически, если только они не копируются в `dist` осознанно

## 12. Класс 2. JSON в `src/lib/*/data/json`

Это уже библиотечный JSON-слой внутри доменов.

Почти у каждого домена есть структура вида:

- `data/json/class/*`
- `data/json/const/*`
- `data/json/function/*`
- `data/json/interface/*`
- `data/json/type/*`

Что это означает:

- такие JSON уже ближе к packaged library content
- они описывают библиотечные сущности по доменам и кластерам
- при упаковке они, как правило, попадают в `dist/<domain>/data/json/...`

С практической точки зрения:

- это не просто служебные JSON для разработчика
- это уже часть библиотечной data-поверхности, если она попадает в `dist`

## 13. Класс 3. JSON как производные отчеты и метаописания

Есть JSON, которые нужны не для конечного пользователя библиотеки, а для:

- inventory
- dependency analysis
- SAMO-отчетов
- внутренних справочников
- генераторов

Такие JSON особенно подозрительны с точки зрения npm-пакета:

- если они попали в `dist`, надо отдельно решать, являются ли они частью контракта
- если они не нужны runtime-пользователю, их лучше не публиковать

## 14. Что я знаю про JSON на текущем этапе

### Знаю точно

- JSON в проекте массово используется
- JSON существует и в tooling-слое, и в библиотечном слое
- JSON присутствует почти во всех доменах библиотеки
- часть JSON уже попадает в `dist`

### Что это значит для релиза

- нужно отдельно решать, какие JSON являются публичными библиотечными данными
- нужно отдельно решать, какие JSON являются только внутренними артефактами
- без такого разделения npm-пакет будет разрастаться и шуметь

## 15. Практический вывод по проекту

Сейчас `stylist-svelte` это не просто библиотека и не просто сайт. Это совмещенный репозиторий из трех слоев:

- слой библиотеки: `src/lib` -> `dist`
- слой сайта/демо/серверных маршрутов: `src/routes` + `static`
- слой tooling/generation/analysis: `scripts`, `docs`, `adr`, `src/test`

Главный релизный вывод:

- npm-пакет контролируется через `dist`
- сайт контролируется через `src/routes` и `static`
- JSON требует отдельной publish-политики, иначе в пакет будет попадать слишком много служебного материала

## 16. Что стоит сделать следующим шагом

1. Отдельно составить список того, что именно сейчас попадает в `dist`, но не должно публиковаться.
2. Отдельно классифицировать JSON на:
   - runtime-public
   - build-time internal
   - analytical/internal
3. Отдельно решить статус доменов `development`, `domain`, `playground`:
   - это часть публичной библиотеки
   - или это внутренний/site-only слой
