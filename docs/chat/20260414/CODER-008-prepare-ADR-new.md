CODER-008-prepare-ADR-new

# ADR-NEW: Нормативная модель SAMO для `stylist-svelte`

**Статус:** draft for review  
**Дата:** 2026-04-15  
**Область действия:** `stylist-svelte/src/lib/**`  
**Назначение:** единый нормативный источник по структуре `domain/cluster/joint/family`

---

## 1. Проблема

Текущая кодовая база `stylist-svelte/src/lib/**` и действующие регламенты (`ADR.md`, `AGENTS.md`, `QWEN.md`) расходятся по фактическому словарю `cluster/joint`.

Выявленные проблемы:
- в коде существуют joint, отсутствующие в регламенте
- в регламенте есть устаревшие или неполные описания
- часть путей нарушает модель `domain/cluster/joint/family`
- терминология `component`, `contract`, `recipe`, `proto` используется не всегда последовательно

Следствие:
- архитектура SAMO не может считаться нормативно стабильной
- автоматическая проверка структуры не может быть полной
- миграция и правовая упаковка проекта опираются на противоречивую базу

---

## 2. Решение

Принять для `stylist-svelte/src/lib/**` строгую морфологическую модель:

```text
<domain>/<cluster>/<joint>/<family>
```

где:
- `domain` — предметная область
- `cluster` — языковая или файловая форма сущности
- `joint` — архитектурная роль сущности внутри cluster
- `family` — конкретное семейство сущности

Нормативная цель:
- любая сущность в `src/lib/**` должна быть адресуема через эту модель
- отклонения допускаются только через новый ADR

---

## 3. Базовые термины

### 3.1 Domain

`domain` — предметная область, в рамках которой группируются сущности одного смыслового поля.

Примеры:
- `architecture`
- `layout`
- `interaction`
- `theme`
- `typography`

### 3.2 Cluster

`cluster` — обязательная языковая или файловая классификация сущности.

Cluster задаёт допустимый вид экспортируемой сущности.

### 3.3 Joint

`joint` — архитектурная роль внутри cluster.

Joint:
- должен иметь поперечно-доменную семантику
- не должен быть доменно-специфичным именем
- не вводится без живого use-case или отдельного ADR

### 3.4 Family

`family` — конкретное семейство или подсемейство сущности.

Допускаются:
- `<family>`
- `<family-trait>`
- `<family-trait-trait>`

---

## 4. Нормативные cluster

В `stylist-svelte/src/lib/**` допускаются только следующие cluster:

| Cluster | Содержимое |
|---------|------------|
| `const` | только `export const` |
| `type` | только `export type` |
| `interface` | только `export interface` |
| `class` | только `export class` |
| `function` | только `export function` |
| `svelte` | Svelte-компоненты и story-файлы |
| `data` | статические данные и ресурсные файлы |

Запрещено:
- вводить новый cluster без ADR
- смешивать несколько языковых ролей в одном cluster

## 5. Нормативные joint

## 5.1 Cluster `const`

Разрешённые joint:
- `enum`
- `value`
- `map`
- `record`
- `struct`
- `config`
- `icon`

Семантика:
- `enum` — закрытые наборы значений и перечислимые бизнес-состояния
- `value` — одиночные скалярные константы
- `map` — явные отображения ключ-значение
- `record` — lookup-таблицы и индексируемые словари
- `struct` — структурные константные объекты
- `configuration` — настройки, defaults, limits, presets
- `icon` — иконочные константы и иконные реестры

## 5.2 Cluster `type`

Разрешённые joint:
- `enum`
- `alias`
- `map`
- `record`
- `struct`
- `configuration`
- `contract`

Семантика:
- `enum` — типы перечислимых значений
- `alias` — псевдонимы типов
- `map` — типы отображений
- `record` — типы lookup-структур
- `struct` — структурные типы данных
- `configuration` — типы конфигурационных объектов
- `contract` — типовые контракты валидации или граничных структур

## 5.3 Cluster `interface`

Разрешённые joint:
- `proto`
- `recipe`
- `contract`
- `record`
- `provider`
- `api`

Семантика:
- `proto` — атомарный поведенческий контракт
- `recipe` — итоговая композиционная схема сборки интерфейса или компонента
- `contract` — граничный или валидационный интерфейсный контракт
- `record` — интерфейс агрегированной записи
- `provider` — интерфейс контекстного поставщика
- `api` — интерфейс интеграционной или прикладной границы

## 5.4 Cluster `class`

Разрешённые joint:
- `manager`
- `object-manager`
- `style-manager`
- `state-manager`

Семантика:
- `manager` — общий управляющий класс домена
- `object-manager` — управление коллекцией однотипных объектов
- `style-manager` — управление стилевыми вычислениями и вариативностью
- `state-manager` — управление сложным состоянием и переходами

## 5.5 Cluster `function`

Разрешённые joint:
- `script`
- `state`
- `hook`
- `transform`
- `test`

Семантика:
- `script` — процедуры и утилиты домена
- `state` — функции реактивного состояния
- `hook` — переиспользуемые реактивные паттерны и binding с runtime
- `transform` — чистые преобразования `A -> B`
- `test` — тестовые сценарии и вспомогательные проверки

## 5.6 Cluster `svelte`

Разрешённые joint:
- `atom`
- `molecule`
- `organism`
- `development`
- `playground`

Семантика:
- `atom`, `molecule`, `organism` — уровни Atomic Design
- `development` — служебные компоненты разработки
- `playground` — компоненты и обвязка демонстрационной среды

`development` и `playground` допускаются как служебные UI-joint проекта и не считаются общим переносимым стандартом SAMO без отдельного решения.

## 5.7 Cluster `data`

Разрешённые joint:
- `json`
- `shader`
- `svg`
- `md`
- `catalog`

Семантика:
- `json` — структурированные данные
- `shader` — `.frag` и `.vert`
- `svg` — векторные ресурсы
- `md` — markdown-документация
- `catalog` — каталогизированные ресурсные наборы

`catalog` допускается как transitional joint до отдельной нормализации data-layer.

## 6. Нормативная модель интерфейсной сборки

Для `interface/**` принимается двухуровневая базовая модель:

```text
proto -> recipe
```

### 6.1 `proto`

`proto` описывает минимальную атомарную способность или поведенческую роль.

Примеры:
- clickable
- selectable
- focusable

### 6.2 `recipe`

`recipe` описывает композицию нескольких `proto` и других совместимых контрактов в единую схему сборки.

`recipe`:
- является основной итоговой формой композиционного интерфейса
- задаёт состав и совместимость
- в типичном случае уже соответствует готовому интерфейсу компонента

### 6.3 `contract`

`contract` не является обязательным третьим уровнем после `recipe`.

`contract` используется только тогда, когда:
- сущность не укладывается в обычный `recipe`
- нужен отдельный граничный, интеграционный или валидационный контракт
- требуется выразить не композицию поведенческих ролей, а специальную форму обязательств

Таким образом:

```text
proto -> recipe
proto -> recipe + optional contract
```

`contract` — это специальное ответвление, а не обязательная ступень конвейера.

### 6.4 Legacy-статус `interface/component`

Пути вида `interface/component/**`, присутствующие в текущем проекте, не считаются нормативным ядром модели SAMO.

На дату настоящего ADR они рассматриваются как legacy или transitional structure и требуют отдельной нормализации:
- либо миграции в `recipe`
- либо миграции в `contract`
- либо отдельного ADR, если будет доказана самостоятельная архитектурная роль `component`

## 7. Assembly direction

Обязательное направление зависимостей:

```text
data -> const -> type -> interface -> class -> function -> svelte
```

Правила:
- обратные зависимости запрещены
- исключения допускаются только отдельным ADR
- при сомнении приоритет имеет более ранний cluster

---

## 8. Правила именования

### 8.1 Общие правила

- один файл экспортирует одну сущность
- один путь соответствует одной архитектурной роли
- top-level необъявленные сущности не допускаются
- joint не должен содержать доменную специфику

### 8.2 Что запрещено на уровне joint

Запрещено считать нормативным joint вида:
- `alert-card`
- `hero-cta`
- `format-date-time`
- и любые иные имена, описывающие конкретное семейство вместо архитектурной роли

Такие имена допустимы только на уровне `family`.

## 10. Правила indexation и audit

### 10.1 Barrel policy

В `stylist-svelte/src/lib/**`:
- нельзя вручную редактировать barrel `index.ts`
- нельзя добавлять ручные re-export layers

### 10.2 Indexation policy

После изменений, влияющих на экспортную структуру, необходимо запускать:

```text
python -u "d:\2026\code\template\stylist\indexation\cli.py"
```

### 10.3 Error checking policy

Для проверки ошибок используется:

```text
python -u "d:\2026\code\template\stylist\errors\cli.py"
```

### 10.4 Audit policy

Целевая модель SAMO считается нормативной только в той степени, в которой она проверяется автоматикой.

`stylist/audit` должен:
- валидировать cluster
- валидировать joint
- обнаруживать отклонения от assembly direction
- строить baseline и progress reports

## 11. Legacy и переходный режим

До полной миграции допускаются transitional paths, если они:
- явно известны
- описаны в migration map
- не расширяются новыми сущностями

Переходные сущности делятся на:
- `legalized` — фактически существуют и теперь приняты в ADR
- `transitional` — временно допускаются до миграции
- `forbidden` — не должны использоваться для новых сущностей

## 12. Что принимается сразу

Настоящим ADR принимаются как нормативные:
- базовая интерфейсная модель `proto -> recipe`
- `interface/contract`
- `interface/record`
- `interface/provider`
- `class/state-manager`
- `const/icon`
- `const/value`
- `type/alias`
- `const/configuration`
- `type/configuration`
- `function/hook`
- `function/transform`
- `data/md`

---

## 13. Что сознательно не принимается этим ADR

Не принимаются как нормативные без отдельного решения:
- `enum -> list`
- `single`
- `function/method`
- `class/director`
- `class/builder`
- `function/logic`
- обязательная трехуровневая цепочка `proto -> recipe -> component`

Рассматриваются как legacy/transitional, но не как нормативное ядро:
- `interface/component`

Причины:
- нет достаточного консенсуса
- нет живого устойчивого use-case
- высок риск терминологического размывания модели

## 14. План миграции

### Шаг 1
- утвердить этот ADR
- синхронизировать `AGENTS.md`
- синхронизировать `QWEN.md`

### Шаг 2
- провести baseline inventory структуры
- собрать migration map
- выделить critical violations
- отдельно инвентаризировать все `interface/component/**` как legacy-кандидаты

### Шаг 3
- обновить `stylist/audit` под новый словарь
- обновить отчёты baseline/progress

### Шаг 4
- мигрировать legacy-пути пакетами
- после каждого пакета запускать:
  - `indexation`
  - `errors`
  - `audit`

## 15. Критерии готовности

SAMO считается приведённой к нормативному состоянию, когда одновременно выполнено:
- `ADR-new`, `AGENTS.md`, `QWEN.md` не противоречат друг другу
- все новые сущности создаются только в рамках утверждённого словаря
- `audit` автоматически распознаёт и классифицирует все нормативные cluster и joint
- количество critical structural violations сведено к нулю
- legacy-отклонения либо мигрированы, либо явно помечены как transitional

---

## 16. Решение по правовой и коммерческой линии

Настоящий ADR не является патентным документом и не содержит юридических claims.

Настоящий ADR создаёт нормативную базу для последующих артефактов:
- commercial matrix
- IP matrix
- novelty map
- claim candidates

Коммерческая и патентная упаковка SAMO допускается только после появления:
- baseline метрик
- измеримого technical effect
- prior art map
- нормализованной структуры проекта

---

## 17. Итог

SAMO в рамках `stylist-svelte` определяется как:
- строго адресуемая морфологическая архитектура
- с фиксированными `cluster/joint`
- с нормативной базовой моделью интерфейсной сборки `proto -> recipe`
- с `contract` как специальным интерфейсным joint, а не обязательным уровнем конвейера
- с автоматической проверкой структуры через tooling

Этот ADR фиксирует норму. Все дальнейшие расширения модели допускаются только через следующий ADR.
