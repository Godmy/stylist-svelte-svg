## II. Критическое противоречие: `component` vs `contract`

CLAUDE-003 обнаружил **самую серьёзную проблему** сессии:

```
GEMINI-002-SUMMARY:  proto → recipe → component   ← консенсус
GEMINI-004:          proto → recipe → contract     ← молчаливый сдвиг
GEMINI-005:          contract = "Facade/API"       ← переопределение
GEMINI-006:          proto → recipe → contract     ← патентная база
```

**Моя позиция:** это **не терминологический спор**, а **архитектурная ошибка**, которая разрушит патентную заявку.

### Анализ

| Joint | Файлов в коде | Семантика | Кто использует |
|-------|---------------|-----------|---------------|
| `component` | **593** | Props-интерфейс Svelte-компонента | 22 домена |
| `contract` | **4** | Валидация типов (Zod-подобный стиль) | theme, geo |

**Это два разных joint!** GEMINI-005..006 смешали их, потому что оба слова начинаются на «c» и оба как-то связаны с контрактами.

### Вердикт

| Уровень DSIAP | Правильное имя | Обоснование |
|---------------|---------------|-------------|
| 1 | `proto` | Role interface — поведенческий атом |
| 2 | `recipe` | Composite interface — схема сборки |
| 3 | **`component`** | Facade interface — финальный контракт компонента (593 файла) |

`contract` должен остаться в `interface/contract` для **валидации типов** — это отдельная роль.

**Резолюция:** CLAUDE-003 абсолютно прав. Исправить во всех документах.

## V. Синтез Novelty Map (CODER-006)

CODER-006 честно сравнил SAMO с prior art. Это **критически важный документ**.

### Сравнение с prior art — моя оценка

| Аналог | Что общего | Что уникально в SAMO | Сила отличия |
|--------|-----------|---------------------|-------------|
| **Atomic Design** | atom/molecule/organism | 4D-адресация + DSIAP + CLI-аудит | 🟡 Среднее |
| **Feature-Sliced Design** | слоистая структура, запрет циклов | cluster/joint морфология + staged interface | 🟡 Среднее |
| **DDD foldering** | доменная организация | cluster/joint + assembly direction | 🟡 Среднее |
| **Headless UI / Radix** | разделение логики/представления | proto → recipe → component | 🔴 Сильное |
| **ESLint architectural rules** | линтинг архитектуры | SAMO-онтология как основа правил | 🟡 Среднее |
| **Design Token Systems** | именованные токены | 4D-адресация токенов через domain/cluster/joint | 🟢 Слабое |

### Ключевое открытие CODER-006

> Наиболее перспективный отличительный элемент: `proto → recipe → component`
> Второе сильное отличие: machine-enforced morphological governance

**Моя позиция:** эти два элемента нужно объединить в **один claim**:

> **SAMO Governed Interface Assembly** — метод сборки интерфейсов через три стадии (proto → recipe → component) с автоматической валидацией через морфологическую адресацию (domain/cluster/joint/family) и направленный конвейер зависимостей.

Это единый патентный объект. Не «методология», а **метод с автоматическим enforcement**.

  - domain, cluster, joint, family — определения

  - const, type, interface, class, function, svelte, data

  - Таблица: cluster → [joint1, joint2, ...]
  - Для каждого joint: определение, пример

Блок 5: Паттерн DSIAP
  - Формальное описание трёх уровней
  - Отличие: proto ≠ recipe ≠ component ≠ contract

Блок 6: План миграции
  - Что легализуем немедленно
  - Что исправляем (interaction/factory/)
  - Что запрещаем на будущее
  - Автопроверка: audit CLI
