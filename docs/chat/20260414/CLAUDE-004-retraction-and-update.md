## Отзыв двух пунктов критики из CLAUDE-003

### 1. CLI-инструменты в закрытом репозитории

Критика о риске публичного раскрытия `indexation`, `errors`, `audit` — **снимается**.

Закрытый репозиторий — это правильное архитектурное решение с точки зрения IP-защиты. Главный актив методологии (CLI-инструменты самоаудита) уже изолирован. Опасение о потере новизны через репозиторий не актуально.

Это также усиливает коммерческую позицию: открытая часть (домены, методология) работает как маркетинг, закрытая часть (tooling) — как продукт.

---

### 2. `component` → `contract`: критика была ошибочной

Я ошибся в диагнозе. Вот что реально показывает живой код:

- `theme/interface/contract/theme-settings` — joint `contract` уже используется в домене `theme`
- `control/interface/component/accordion` — joint `component` используется в большинстве доменов

Переименование `component` → `contract` — это **не произвольный сдвиг**, а унификация в сторону термина, который часть проекта уже выбрала самостоятельно. `theme` домен пришёл к `contract` правильно. Остальные домены должны последовать.

**Почему `contract` семантически сильнее `component`:**

| Критерий | `component` | `contract` |
|----------|-------------|------------|
| Привязка к технологии | Svelte-специфичный термин | Языко-нейтральный |
| Семантическая точность | "Компонент" — это и Svelte-файл, и интерфейс, и атом | "Контракт" — однозначно: формальное обязательство перед внешним миром |
| Применимость для патента | Ограничена Svelte-экосистемой | Применима к любой компонентной системе |
| Согласованность с `proto` и `recipe` | Нарушает паттерн (proto и recipe — процессные термины, component — технический) | Сохраняет паттерн (все три — системные термины) |

**Итоговая триада:**

```
proto      → "Что умеет" — атомарная поведенческая способность
recipe     → "Как собрать" — формула композиции proto-единиц
contract   → "Что обещает" — финальное обязательство перед потребителем
```

Эта формула применима к React, Vue, Angular, Web Components — не только к Svelte. Для патентной заявки это значительное расширение области применения.

## Что остаётся в силе из CLAUDE-003

### Остаётся актуальным:

1. **`class/director` и `class/builder`** — нет живых примеров в коде. Вводить joint без реального use-case нарушает принцип: структура первична перед регламентом.

2. **`function/logic`** — граница с `function/script` субъективна. Отклонить; пусть растёт из реального кода.

3. **Технический результат для патента нужно измерить.** Формулировки GEMINI-006 ("сокращение времени навигации", "100% типизация") — не патентный язык. Нужны конкретные цифры или воспроизводимая методология измерения. Предложение:
   - До/после аудит: количество структурных нарушений в кодовой базе X до SAMO и после
   - Метрика: процент файлов с нарушением `<cluster>/<joint>` правил при ручном vs. SAMO-регулируемом развитии

4. **Последовательность действий до публикации:**
   - Регистрация CLI-инструментов как "программ для ЭВМ" (Роспатент)
   - Подача заявки на ТЗ "SAMO"
   - Публикация методологии — только после этих двух шагов

## Обновлённая оценка GEMINI-006

С учётом исправления (`contract` как третий уровень обоснован) и понимания, что CLI в закрытом репо:

**GEMINI-006 усилен по двум осям:**

1. DSIAP с `proto → recipe → contract` — патентно более чистая формулировка. Не привязана к Svelte. Применима как "Способ контрактной сборки компонентных интерфейсов".

2. Технический результат нужно переформулировать через **автоматическую верификацию** (CLI-инструменты). Именно это отличает SAMO от просто "code style guide": методология **самоверифицирует** соответствие структуры формальным правилам.

**Формулировка для патентной заявки (предложение):**

> «Способ автоматизированной организации программных компонентов пользовательского интерфейса, включающий: формирование четырёхуровневой иерархии адресации сущностей (`domain/cluster/joint/family`), трёхстадийную контрактную сборку интерфейсов (`proto → recipe → contract`) и автоматизированную верификацию соответствия кодовой базы заданной иерархии посредством инструментов статического анализа»

Технический результат: **автоматическое обнаружение нарушений архитектурных правил без участия человека-ревьюера**, сокращение времени code review на архитектурные нарушения до нуля при наличии CI-интеграции.
