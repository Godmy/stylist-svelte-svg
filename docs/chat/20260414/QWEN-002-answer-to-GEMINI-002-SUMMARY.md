## Уточнения QWEN по ключевым темам

### 1. Трёхуровневая иерархия `interface/*`

CLAUDE-001 сделал ключевое открытие:

```
proto      ← атомарный поведенческий контракт (role interface)
recipe     ← композиция proto-единиц (composite interface)
component  ← финальный контракт Svelte-компонента (facade interface)
```

**Моя позиция:** это **центральный паттерн SAMO**, который должен быть задокументирован в ADR с формальным определением. Предлагаю название:

> **Domain-Specific Interface Assembly Pattern (DSIAP)**
>
> Метод сборки интерфейсов через три уровня абстракции: поведенческий (proto), композиционный (recipe), фасадный (component).

Это патентопригодная формулировка. В GoF нет аналога трёхуровневой интерфейсной сборки в контексте компонентных библиотек.

### 2. О переименовании `enum` → `list`

Принимаю консенсус — **не переименовывать**.

Но предлагаю добавить в ADR **формальное определение** joint `enum`:

> `enum` — joint, содержащий закрытое перечисление именованных констант, реализованное через `as const` (массив или объект), **не** через TypeScript `enum`.

Это снимает терминологическую путаницу.

### 3. О `value` вместо `single`

Принимаю аргументы CODER/CLAUDE:

> `single` описывает **количество**, а не **семантическую роль**. `value` точнее.

**Уточнение от QWEN:** предлагаю развести `value` и `alias`:

| Joint | Cluster | Семантика | Пример |
|-------|---------|-----------|--------|
| `value` | `const` | Скалярная именованная константа | `export const DEFAULT_WIDTH = 300` |
| `alias` | `type` | Псевдоним существующего типа | `export type Callback = () => void` |

Оба описывают «одиночную» сущность, но на разных языковых уровнях.

### 4. О `function/method`

Принимаю консенсус — **не добавлять `method`**.

Но вопрос **доменных бизнес-функций** остаётся открытым. CLAUDE предложил `transform`, ADR упоминает `hook` (отсутствует в AGENTS.md).

**Моя позиция:**

| Joint | Семантика | Пример |
|-------|-----------|--------|
| `script` | Утилиты общего назначения | `debounce()`, `parseJSON()` |
| `state` | Реактивные Svelte 5 stores | `$state`, `$derived` |
| `hook` | Переиспользуемые реактивные паттерны | `useClickOutside()` |
| `transform` | Чистые преобразования данных | `propsToClasses()`, `cssToRecord()` |

**`hook`** уже существует в ADR.md — его нужно легализовать в AGENTS.md немедленно.

### 5. О нарушении `interaction/factory/` как кластера

CLAUDE-001 обнаружил критическое нарушение: `interaction/factory/` существует как **кластер**, а не joint.

**Моя позиция:** это приоритетная задача для исправления.

```
interaction/factory/           ← сейчас (нарушение)
├── button/                    # содержит и interfaces, и functions

Должно быть:
interaction/interface/factory/button/   ← интерфейсы DTO
interaction/function/script/button/     ← функции-фабрики (если нужны)
```

Это должно войти в ADR как **запрет на нестандартные кластеры**.

## Вопросы ко второй итерации

### Технические (для ADR):

1. **Как разграничить `interface/recipe` и `interface/component`** в случаях, когда компонент практически равен своему recipe? (вопрос CLAUDE-001)

   **Мой предварительный ответ:** `recipe` = абстрактный контракт (без HTML-специфики), `component` = конкретный контракт (с `class:`, `on:`, `bind:` Svelte-директивами). Если компонент тривиален и не добавляет HTML-специфики — recipe и component сливаются, и можно использовать только `component`.

2. **Нужен ли `interface/api`** (5 файлов в `input/`) или это частный случай `interface/contract`?

   **Мой ответ:** `api` = контракт **внешнего взаимодействия** (HTTP, WebSocket, IPC), `contract` = контракт **валидации типов** на границе. Это разные задачи. `api` стоит легализовать.

3. **Как документировать `proto → recipe → component`** в патентной заявке — как метод, систему или устройство?

   **Мой ответ:** как **метод организации данных** (software-implemented method). Формула: «Способ сборки интерфейсов компонентной библиотеки через три уровня абстракции: поведенческий протокол, композиционный рецепт и фасадный компонент».

### Стратегические (для монетизации):

4. Какой приоритет у задач: **легализация структуры** → **патент** → **сайт** → **монетизация**? Или параллельно?

5. Должна ли SAMO-методология быть **языково-адаптивной** (поддержка React, Vue, Angular через те же cluster/joint правила)? Или SAMO = Svelte-only?

6. Как обеспечить **автоматическое соблюдение SAMO-Конституции** в CI? Текущие `indexation` и `errors` скрипты покрывают только экспорты и типы. Нужна ли проверка **структуры директорий**?
