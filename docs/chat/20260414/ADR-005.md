### 1.2 Нарушение архитектуры: нестандартные кластеры

Обнаружено нарушение кластерного контракта:

```
interaction/factory/          ← кластер вне регламента
├── button/                   # содержит одновременно интерфейсы и функции
├── toggle/
└── ...
```

Содержимое одновременно ссылается на `interface` и `function` уровни, нарушая:
- Cluster Policy (кластер должен содержать один тип сущностей)
- Assembly Direction (направление зависимостей размывается)

### 1.3 Доменно-специфичные joint

В живом коде обнаруены joint-имена, которые являются доменными семействами, а не универсальными соединениями:

```
commerce/const/alert-card/        ← joint = "alert-card" (только commerce)
commerce/const/return-policy/     ← joint = "return-policy" (только commerce)
marketing/type/hero-cta/          ← joint = "hero-cta" (только marketing)
localization/function/format-date-time/  ← joint = "format-date-time" (только localization)
```

Это размывает универсальный словарь SAMO. Такие имена должны быть `family`, а не `joint`.

## Блок 2: Термины SAMO

### 2.1 Четырёхмерная адресация

Каждая программная сущность адресуется четырьмя координатами:

```
src/lib/<domain>/<cluster>/<joint>/<family>/index.ts
```

| Координата | Название | Что описывает | Пример |
|-----------|----------|---------------|--------|
| 1-я | **Domain** | Предметная область | `interaction`, `theme`, `commerce` |
| 2-я | **Cluster** | Технологическая форма (тип языка) | `const`, `type`, `interface`, `class`, `function`, `svelte`, `data` |
| 3-я | **Joint** | Архитектурная роль сущности | `enum`, `proto`, `recipe`, `manager`, `state` |
| 4-я | **Family** | Конкретная функциональная принадлежность | `clickable`, `theme-mode`, `accordion` |

### 2.2 DSIAP — Domain-Specific Interface Assembly Pattern

Метод сборки интерфейсов через три стадии абстракции:

```
┌─────────────────────────────────────────────────────────┐
│                    DSIAP                                 │
│                                                          │
│  Stage 1: proto (Role Interface)                         │
│  ─────────────────────────────────────                    │
│  Атомарный поведенческий контракт.                       │
│  Описывает ОДНУ способность объекта.                     │
│  Пример: IClickable, IFocusable, IHoverable             │
│  Joint: interface/proto/<family>                         │
│                                                          │
│  Stage 2: recipe (Composite Interface)                   │
│  ─────────────────────────────────────                    │
│  Композиция proto-единиц.                                │
│  Описывает ЧЕРТЁЖ сборки сущности.                       │
│  Пример: AccordionRecipe = ISized + IScrollable + ICollapsible │
│  Joint: interface/recipe/<family>                        │
│                                                          │
│  Stage 3: component (Facade Interface)                   │
│  ─────────────────────────────────────                    │
│  Финальный контракт Svelte-компонента.                   │
│  Включает HTML-специфику (class:, on:, bind:).          │
│  Пример: AccordionProps = AccordionRecipe + DOM-атрибуты │
│  Joint: interface/component/<family>                     │
└─────────────────────────────────────────────────────────┘
```

**Важно:** `contract` — это НЕ третий уровень DSIAP. `contract` — joint для **валидации типов** на границе (Zod-подобный стиль), используемый в `theme/interface/contract` и `geo/interface/contract`.

### 2.3 Assembly Direction (Направленный конвейер)

Строгое однонаправленное направление импортов:

```
data → const → type → interface → class → function → svelte
```

Запрещены:
- Обратные зависимости (svelte → const допустимо, const → svelte — нет)
- Горизонтальные зависимости одного уровня без явного ADR
- Циклические зависимости любого вида

## Блок 3: Обязательные кластеры

Каждый домен может содержать только следующие кластеры:

| Кластер | Содержимое | TypeScript-конструкция |
|---------|-----------|----------------------|
| `const` | Только константы | `export const X = ...` |
| `type` | Только типы | `export type T = ...` |
| `interface` | Только интерфейсы | `export interface I { ... }` |
| `class` | Только классы | `export class C { ... }` |
| `function` | Только функции | `export function f() { ... }` |
| `svelte` | Svelte-компоненты | `*.svelte` файлы |
| `data` | Статические файлы | `*.json`, `*.svg`, `*.md`, `*.frag`, `*.vert` |

**Запрет:** создание кластеров вне данного перечня без нового ADR.

## Блок 4: Допустимые joint по каждому кластеру

### 4.1 Кластеры `const` и `type`

| Joint | Семантика | Пример | Только для кластера |
|-------|-----------|--------|-------------------|
| `enum` | Закрытое перечисление (`as const`) | `const/enum/theme-mode/` → `export const THEME_MODES = ['light', 'dark'] as const` | — |
| `value` | Скалярная именованная константа | `const/value/default-width/` → `export const DEFAULT_WIDTH = 300` | `const` |
| `alias` | Псевдоним существующего типа | `type/alias/callback/` → `export type Callback = () => void` | `type` |
| `struct` | Структурный объект (DTO, Value Object) | `const/struct/point/` → `export const POINT = { x: 0, y: 0 }` | — |
| `record` | Lookup-таблица (ключ→значение) | `const/record/size/` → `export const SIZE_MAP = { sm: 0.8, md: 1, lg: 1.2 }` | — |
| `map` | Словарь с явными ключами | `const/map/style/` → `export const STYLE_MAP = new Map<string, StyleValue>()` | — |
| `icon` | Реестр иконочных констант | `const/icon/arrow/` → `export const ARROW_ICON = 'arrow-right'` | — |
| `configuration` | Конфигурационный объект (параметры, defaults, presets) | `const/configuration/canvas-config/` → `export const CANVAS_CONFIG = { width: 800, height: 600 }` | — |

**Определение `enum`:** joint `enum` содержит закрытые перечисления, реализованные через `as const` (массивы и объекты). Это **не** TypeScript `enum`-типы. TypeScript `enum`-типы (если используются) размещаются в `type/enum`.

**Различие `struct` vs `configuration`:**
- `struct` описывает **данные** (форму объекта)
- `configuration` описывает **параметры поведения** системы

### 4.2 Кластер `interface` — DSIAP

| Joint | Стадия DSIAP | Семантика | Пример |
|-------|-------------|-----------|--------|
| `proto` | Stage 1 | Role Interface — атомарный поведенческий контракт | `interface/proto/clickable/` → `export interface ClickableProps { onClick?: () => void }` |
| `recipe` | Stage 2 | Composite Interface — композиция proto-единиц | `interface/recipe/accordion/` → `export interface AccordionRecipe extends ISized, IScrollable, ICollapsible {}` |
| `component` | Stage 3 | Facade Interface — финальный контракт Svelte-компонента | `interface/component/button/` → `export interface ButtonProps extends IClickable, IStylable { label: string }` |
| `contract` | — | Boundary Validation — валидация типов на границе | `interface/contract/theme/` → `export interface ThemeContract { mode: ThemeMode; colors: ColorMap }` |
| `record` | — | Interface-запись (агрегат данных) | `interface/record/event-log/` → `export interface EventLogRecord { timestamp: number; event: string }` |
| `factory` | — | DTO для фабричных функций | `interface/factory/accordion/` → `export interface AccordionFactoryInput { items: AccordionItem[]; defaultOpen?: number }` |
| `provider` | — | Контекстный интерфейс (Svelte Context) | `interface/provider/theme/` → `export interface ThemeProvider { getTheme: () => Theme; setTheme: (t: Theme) => void }` |

**Правило разграничения:**

| Если интерфейс... | Используй joint |
|-------------------|-----------------|
| Описывает одну способность | `proto` |
| Описывает сборку из proto | `recipe` |
| Описывает props Svelte-компонента | `component` |
| Описывает валидацию на границе | `contract` |
| Описывает входные данные фабрики | `factory` |
| Описывает контекстный провайдер | `provider` |
| Описывает запись-агрегат | `record` |

### 4.3 Кластер `class`

| Joint | Семантика | Пример |
|-------|-----------|--------|
| `manager` | Операционное управление жизненным циклом | `class/manager/theme/` → управляет темами |
| `object-manager` | Репозиторий — управление коллекцией однотипных объектов | `class/object-manager/node/` → узлы canvas |
| `style-manager` | Декоратор стилей — выделение CSS-логики из UI | `class/style-manager/container/` → CSS-классы контейнеров |
| `state-manager` | Управление состоянием (навигация, роутер) | `class/state-manager/router/` → состояние роутера |

### 4.4 Кластер `function`

| Joint | Семантика | Тип функции | Пример |
|-------|-----------|-------------|--------|
| `script` | Утилиты общего назначения | Чистые функции, процедуры | `script/parse-date/` → `parseDate(str: string): Date` |
| `state` | Реактивное состояние (Svelte 5 Runes) | Функции, возвращающие `$state` | `state/draggable/` → `useDraggable()` |
| `hook` | Реактивные паттерны жизненного цикла | Подписки, эффекты | `hook/click-outside/` → `useClickOutside()` |
| `transform` | Чистые преобразования данных | `A → B` без побочных эффектов | `transform/props-to-classes/` → `propsToClasses(props)` |
| `test` | Тестовые сценарии | Тест-функции | `test/theme-switcher/` → тесты |

**Разграничение:**

| Если функция... | Используй joint |
|-----------------|-----------------|
| Утилита без привязки к домену | `script` |
| Работает с `$state` / Svelte Runes | `state` |
| Подписывается на DOM/лifecycle | `hook` |
| Преобразует данные без побочных эффектов | `transform` |
| Тестирует | `test` |

### 4.5 Кластер `svelte`

| Joint | Семантика | Пример |
|-------|-----------|--------|
| `atom` | Минимально неделимый UI-контракт | `svelte/atom/button/` |
| `molecule` | Функциональная композиция атомов | `svelte/molecule/form-field/` |
| `organism` | Автономный высокоуровневый блок | `svelte/organism/navigation-bar/` |

### 4.6 Кластер `data`

| Joint | Тип контента | Пример |
|-------|-------------|--------|
| `json` | Структурированные статические данные | `data/json/locale/` → `ru.json` |
| `svg` | Векторные изображения | `data/svg/icons/` → `arrow.svg` |
| `md` | Текстовый контент | `data/md/readme/` → `index.md` |

## Блок 5: Паттерн DSIAP — формальное описание

### 5.1 Определение

**DSIAP** (Domain-Specific Interface Assembly Pattern) — метод синтеза программных интерфейсов через три стадии абстракции:

1. **Proto** — атомарный поведенческий контракт. Минимальный набор свойств, определяющий одну способность объекта. Реализует Interface Segregation Principle (ISP из SOLID).

2. **Recipe** — композиционный контракт. Объединяет несколько proto-единиц в связную структуру. Рецепт не является финальным продуктом — он определяет правила взаимодействия признаков.

3. **Component** — фасадный контракт. Единственная точка входа для внешних систем. Инкапсулирует сложность рецепта и proto-признаков, добавляя платформенную специфику (HTML/Svelte-директивы).

### 5.1 Формальные свойства

| Свойство | Описание |
|----------|----------|
| **Однонаправленность** | `proto → recipe → component`. Обратные зависимости запрещены |
| **Композиционность** | Recipe расширяет один или более Proto. Component расширяет один Recipe |
| **Инкапсуляция** | Внешний потребитель видит только Component, не Proto и не Recipe |
| **Повторное использование** | Один Proto может входить в несколько Recipe. Один Recipe — в несколько Component |

### 5.2 Отличие от `contract`

| Joint | Назначение | Пример содержимого |
|-------|-----------|-------------------|
| `component` | Props-интерфейс Svelte-компонента | `ButtonProps { label, onClick, disabled }` |
| `contract` | Валидация типов на границе | `ThemeContract { validate(mode): boolean }` |

`contract` — это НЕ стадия DSIAP. Это отдельный joint для пограничной валидации.

### 5.3 Отличие от существующих паттернов

| Паттерн | Аналог в SAMO | Отличие |
|---------|---------------|---------|
| Atomic Design (Brad Frost) | `svelte/atom\|molecule\|organism` | SAMO расширяет вниз до `const/type/interface/class/function` |
| Compound Components (React) | `interface/recipe` | SAMO формализует три стадии, а не только композицию |
| Headless UI | `proto` + `component` | SAMO добавляет `recipe` как промежуточную стадию |
| Feature-Sliced Design | `domain` + Assembly Direction | SAMO добавляет `cluster/joint` морфологию |
| Design Token System | `const/enum\|value\|icon` | SAMO добавляет 4D-адресацию токенов |

## Приложение C: Глоссарий терминов

| Термин | Определение |
|--------|------------|
| **SAMO** | SOLID + Atomic Design + Morphological + Organization |
| **DSIAP** | Domain-Specific Interface Assembly Pattern |
| **Domain** | Предметная область (28 доменов в `stylist-svelte`) |
| **Cluster** | Технологическая форма сущности (тип языка) |
| **Joint** | Архитектурная роль сущности внутри кластера |
| **Family** | Конкретная функциональная принадлежность |
| **Family-Trait** | Уточнение семейства (например, `theme-mode` от `theme`) |
| **Assembly Direction** | Однонаправленный конвейер зависимостей |
| **Barrel** | `index.ts` с re-export, генерируется автоматически |
| **Proto** | Атомарный поведенческий контракт (Stage 1 DSIAP) |
| **Recipe** | Композиция proto-единиц (Stage 2 DSIAP) |
| **Component** | Финальный контракт Svelte-компонента (Stage 3 DSIAP) |
| **Contract** | Валидация типов на границе (НЕ стадия DSIAP) |
