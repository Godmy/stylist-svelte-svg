## NOVELTY MAP FOR SAMO

Цель документа:
- не доказать новизну декларативно
- а честно показать, где у SAMO возможна новизна, а где пока только комбинация известных практик

## 1. Базовый тезис

Если SAMO описывать как:
- "иерархия папок"
- "правила именования"
- "Atomic Design + SOLID"

то новизна слабая.

Если SAMO описывать как связку:
- морфологической адресации сущностей
- типизированной иерархии ролей
- контрактной интерфейсной сборки
- автоматической генерации индексов
- автоматического аудита нарушений

то появляется шанс на более сильную дифференциацию.

## 2. SAMO vs Atomic Design

### Что у Atomic Design уже есть
- уровни `atom`, `molecule`, `organism`
- идея композиции UI из простых элементов

### Что у SAMO добавляется
- уровни ниже UI: `const`, `type`, `interface`, `class`, `function`
- единая морфологическая адресация для всех сущностей домена
- машинно-проверяемый assembly direction
- разделение интерфейсной сборки `proto -> recipe -> component`

### Вывод
- SAMO не заменяет Atomic Design
- SAMO расширяет его вниз, в сторону типизации, контрактов и governance

Потенциальная новизна:
- не в `atom/molecule/organism`
- а в полном конвейере от константы до UI-контракта

## 3. SAMO vs Feature-Sliced Design

### Что у FSD уже есть
- структурирование по слоям и ответственности
- борьба с хаотическими импортами
- управляемые зависимости

### Что у SAMO отличается
- адресация идёт через `domain/cluster/joint/family`, а не через feature/widget/entity/process/shared
- более жёсткая морфологическая типизация внутри домена
- сильнее выражен language-level split
- есть отдельная контрактная модель интерфейсов

### Вывод
- SAMO не выглядит принципиально новым только на уровне "слоистой структуры"
- отличимость появляется на уровне `joint` и DSIAP

## 4. SAMO vs DDD foldering / domain-oriented code organization

### Что уже известно в DDD-подходах
- организация по доменам
- ubiquitous language
- bounded contexts

### Что у SAMO отличается
- домен не единственная координата
- вводится второй и третий измеритель: `cluster` и `joint`
- структура ограничивает не только тему, но и форму программной сущности
- правила поддерживаются автоматическим tooling

### Вывод
- доменная часть сама по себе не нова
- потенциальная новизна в многомерной типизированной адресации и автоматическом enforcement

## 5. SAMO vs contract-first UI systems

### Что уже встречается
- props interfaces
- UI contracts
- schema-driven components

### Что у SAMO отличается
- разложение интерфейсов на три архитектурные роли:
  - `proto`
  - `recipe`
  - `component`
- это не просто "props type", а staged interface assembly

### Вывод
- именно здесь находится одно из самых сильных отличий
- но это надо ещё формализовать и проверить на prior art

## 6. SAMO vs codegen + lint + static analysis

### Что уже известно
- линтеры
- custom ESLint rules
- code generators
- архитектурные анализаторы

### Что у SAMO отличается
- rules привязаны к морфологической модели
- индексирование и аудит завязаны на структуру `domain/cluster/joint/family`
- архитектурные отклонения можно выражать как нарушения онтологии, а не только синтаксиса

### Вывод
- tooling само по себе не ново
- новизна может быть в том, что toolchain обслуживает строго определённую онтологию

## 7. Где у SAMO потенциально сильная новизна

### Кандидат A: Morphological Addressing Model
- каждая сущность адресуется по четырём координатам
- координаты одновременно описывают домен, языковую форму, архитектурную роль и семейство

### Кандидат B: DSIAP
- staged interface assembly:
  - `proto` как атомарная способность
  - `recipe` как схема композиции
  - `component` как финальный boundary contract

### Кандидат C: Governed Assembly Pipeline
- фиксированное направление зависимостей
- автоматически проверяемое tooling
- индексирование и аудит как часть архитектурного режима

### Кандидат D: Self-analyzing morphological governance
- методология не только описывает структуру
- она сама себя проверяет через `indexation`, `errors`, `audit`

Из этих четырёх кандидатов сильнейшим на сегодня выглядит `B + C`, а не просто словарь папок.

## 8. Где новизна слабая или сомнительная

- сами названия `domain`, `cluster`, `joint`, `family`
- Atomic Design слой
- идея запрета циклических зависимостей
- разбиение на `const/type/interface/class/function`
- наличие audit CLI как такового

Это всё важно для системы, но не выглядит как сильная самостоятельная новизна.

## 9. Что нужно сделать для честной novelty preparation

### Подготовить таблицу сравнения минимум с:
- Atomic Design
- Feature-Sliced Design
- DDD-style frontend organization
- design token systems
- contract-first component libraries
- architecture linting / static governance tools

### Для каждого сравнения ответить:
- какой признак у них уже есть
- какой признак у SAMO новый
- какой эффект даёт именно комбинация признаков

### Отдельно выделить:
- обязательные признаки метода
- необязательные расширения
- признаки, которые не стоит включать в claims

## 10. Предварительный честный вывод

На сегодня корректная позиция такая:

- SAMO уже выглядит как интересная и сильная композиция известных практик
- наиболее перспективный отличительный элемент: `proto -> recipe -> component`
- второе сильное отличие: machine-enforced morphological governance
- слабейшая линия для claims: просто иерархия папок и naming convention

## Итог

SAMO не стоит пока позиционировать как "очевидно патентно-новую систему".

Корректнее позиционировать так:
- есть сильная архитектурная методология
- есть гипотеза о новизне на уровне DSIAP и governed assembly pipeline
- требуется formal prior art review и claim engineering
