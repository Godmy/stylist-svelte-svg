### 1.2 Семантическая проблема `proto`

`interface/proto` содержал два принципиально разных вида интерфейсов:
- поведенческие способности: `IClickable`, `IFocusable`, `IDraggable`
- места содержимого: `ILabelSlot`, `IIconSlot`, `IBadgeSlot`

Смешение этих видов в одном joint делало `proto` семантически размытым и затрудняло автоматическую классификацию.

### 1.3 Ложный третий уровень `component`

Практика показала (на примере `ButtonRecipe`): `recipe` является финальной точкой интерфейсной сборки и полностью готов к использованию как props Svelte-компонента. `interface/component` создавал избыточный слой без функционального смысла.

```ts
// ButtonRecipe — финальный интерфейс, дополнительный слой не нужен
export interface ButtonRecipe
    extends StructIntersectAll<[
        LabelRecipe,
        IIconSlot,
        IClickable,
        IFocusable,
        ISizable,
        IBadgeSlot
    ]>
{}
```

## Блок 2: Термины SAMO 2.0

### 2.1 Четырёхмерная адресация

Каждая сущность адресуется:

```
src/lib/<domain>/<cluster>/<joint>/<family>/index.ts
```

| Координата | Название | Что описывает | Пример |
|-----------|----------|---------------|--------|
| 1-я | **Domain** | Предметная область | `interaction`, `theme`, `typography` |
| 2-я | **Cluster** | Технологическая форма | `const`, `type`, `interface`, `class`, `function`, `svelte`, `data` |
| 3-я | **Joint** | Архитектурная роль | `role`, `slot`, `recipe`, `contract`, `manager`, `state` |
| 4-я | **Family** | Функциональная принадлежность | `clickable`, `label`, `button`, `theme-context` |

### 2.2 DSIAP 2.0 — двухстадийная модель интерфейсной сборки

**DSIAP** (Domain-Specific Interface Assembly Pattern) — метод синтеза программных интерфейсов через два уровня абстракции:

```
┌──────────────────────────────────────────────────────────────┐
│                    DSIAP 2.0                                  │
│                                                               │
│  Stage 1: role / slot  (атомарный уровень)                   │
│  ─────────────────────────────────────────                    │
│  role  — поведенческая способность объекта                   │
│          "Что умеет делать?"                                  │
│          Примеры: RoleClickable, RoleFocusable, RoleDraggable │
│          Joint: interface/role/<family>                        │
│                                                               │
│  slot  — место для содержимого или данных                     │
│          "Что принимает как содержимое?"                      │
│          Примеры: SlotLabel, SlotIcon, SlotBadge              │
│          Joint: interface/slot/<family>                        │
│                                                               │
│  Stage 2: recipe  (финальный уровень)                        │
│  ─────────────────────────────────────                        │
│  Композиция role и slot через StructIntersectAll.             │
│  Является готовым интерфейсом props компонента.               │
│  Примеры: ButtonRecipe, AccordionRecipe                       │
│  Joint: interface/recipe/<family>                             │
│                                                               │
│  contract  (опциональное ответвление)                        │
│  ─────────────────────────────────────                        │
│  Межсистемный или интеграционный договор.                     │
│  НЕ является стадией конвейера — только специальные случаи.  │
│  Примеры: ContractThemeContext, ContractCalendarDay           │
│  Joint: interface/contract/<family>                           │
└──────────────────────────────────────────────────────────────┘
```

Формульное обозначение:

```
role + slot  →  recipe
role + slot  →  recipe  (+  contract  — только если нужен межсистемный договор)
```

### 2.3 Полная цепочка сборки компонента

```
interface/role  ┐
interface/slot  ├──→  interface/recipe  ──→  function/state  ──→  svelte/atom|molecule|organism
interface/contract? ┘
```

Каждому `role` и `slot` соответствует свой `function/state`, реализующий runtime-поведение. `recipe` задаёт тип props; `function/state` собирает поведение; `svelte`-компонент потребляет готовый state и добавляет разметку.

### 2.4 Assembly Direction

```
data  →  const  →  type  →  interface  →  class  →  function  →  svelte
```

Обратные зависимости и циклы запрещены. Исключения — только через ADR.

## Блок 4: Допустимые joints

### 4.1 Кластер `interface`

| Joint | Семантика | Префикс/суффикс | Статус |
|-------|-----------|-----------------|--------|
| `role` | Поведенческая способность ("что умеет") | `Role*` | ✅ Нормативный |
| `slot` | Место содержимого/данных ("что принимает") | `Slot*` | ✅ Нормативный |
| `recipe` | Финальная сборка интерфейса компонента | `*Recipe` | ✅ Нормативный |
| `contract` | Межсистемный или интеграционный договор | `Contract*` | ✅ Нормативный |
| `record` | Интерфейс-запись (агрегат данных) | `*Record` | ✅ Нормативный |
| `factory` | Входные параметры фабричных функций | `*Factory` | ✅ Нормативный |
| `provider` | Контекстный интерфейс (Svelte Context) | `*Provider` | ✅ Принят |
| `proto` | Атомарный контракт (предшественник role/slot) | `I*` | ⚠️ LEGACY — не использовать для нового кода |
| `component` | Фасадный контракт (устаревший слой) | — | ❌ LEGACY — мигрировать в `recipe` или `contract` |

**Правило выбора joint для `interface`:**

| Если интерфейс описывает... | Joint |
|---------------------------|-------|
| Одну поведенческую способность объекта | `role` |
| Место для содержимого или данных | `slot` |
| Итоговую сборку props компонента | `recipe` |
| Внешний или межсистемный договор | `contract` |
| Входные параметры фабричной функции | `factory` |
| Сущность, передаваемую через setContext | `provider` |
| Агрегированную запись данных | `record` |

**Граница `recipe` и `contract`:**

| Сущность | Joint |
|----------|-------|
| Props кнопки, поля ввода, чекбокса | `recipe` |
| Структура темы (ThemeContext) | `contract` |
| ViewModel для settings-панели | `contract` |
| Данные сущностей (CalendarDay, CalendarEvent) | `contract` |

> Кнопка — recipe. Тема — contract. Если сомневаетесь: может ли эта сущность существовать независимо от конкретного Svelte-компонента? Если да — contract.

## Блок 5: Паттерн StructIntersectAll

### 5.1 Что это

`StructIntersectAll` — type-level утилита, реализующая **рекурсивную свёртку (Fold)** по кортежу типов.

```ts
export type StructIntersectAll<TTypes extends readonly unknown[]> =
    TTypes extends readonly [infer THead, ...infer TTail]
        ? THead & StructIntersectAll<TTail>
        : {};
```

Расположение: `architecture/type/struct/intersect-all/index.ts`

### 5.2 Что это НЕ

| Ошибочное название | Почему неверно |
|-------------------|----------------|
| Builder | Builder — runtime GoF паттерн. StructIntersectAll работает только на уровне типов |
| Factory | Factory создаёт объекты в runtime. StructIntersectAll не существует в runtime |
| Factory Method | Аналогично |
| Static Assembler | Допустимо описательно, но не является GoF паттерном |

Точное описание: **type-level fold** / **recursive intersection utility** / **structural composition helper**.

### 5.3 Использование

```ts
import type { StructIntersectAll } from '$stylist/architecture/type/struct/intersect-all';

export interface ButtonRecipe
    extends StructIntersectAll<[
        LabelRecipe,
        RoleClickable,
        RoleFocusable,
        SlotIcon,
        SlotBadge
    ]>
{}
```

## Блок 7: Нарушения и legacy

### 7.1 `interaction/factory/` — нарушение кластерного контракта

`interaction/factory/` использован как кластер, тогда как `factory` — это joint внутри кластера. Корректный путь:

```
interaction/interface/factory/<family>   ← интерфейс фабрики
interaction/function/script/<family>     ← реализация фабрики
```

### 7.2 Доменно-специфичные joints

Имена вида `alert-card`, `hero-cta`, `format-date-time` на уровне joint — нарушение. Доменные имена допустимы только на уровне `family`.

### 7.3 Запрещённые joint (отклонены)

| Joint | Причина |
|-------|---------|
| `class/director` | Нет live use-case |
| `class/builder` | Нет live use-case; builder — runtime-паттерн |
| `function/logic` | Субъективная граница с `script` |
| `function/method` | ООП-термин в функциональном контексте |

## Приложение B: Глоссарий

| Термин | Определение |
|--------|------------|
| **SAMO** | SOLID + Atomic Design + Morphological + Organization |
| **DSIAP** | Domain-Specific Interface Assembly Pattern (2-stage: role/slot → recipe) |
| **Domain** | Предметная область (`interaction`, `theme`, `commerce`, ...) |
| **Cluster** | Технологическая форма (`const`, `interface`, `function`, ...) |
| **Joint** | Архитектурная роль внутри кластера |
| **Family** | Конкретное семейство сущности |
| **Role** | Поведенческая способность объекта (joint `interface/role`) |
| **Slot** | Место содержимого или данных (joint `interface/slot`) |
| **Recipe** | Финальная интерфейсная сборка компонента |
| **Contract** | Межсистемный или интеграционный договор |
| **Proto** | Legacy joint — предшественник `role` и `slot` |
| **StructIntersectAll** | Type-level fold: рекурсивное пересечение кортежа типов |
| **Assembly Direction** | Однонаправленный конвейер зависимостей: `data → const → ... → svelte` |
| **Barrel** | `index.ts` с re-export, генерируется только автоматически |
