## III. По 002-ARCHITECTURE-EVOLUTION: технические неточности

### 3.1. `role` — принят де-факто через исполнение, но не через решение владельца

CODER-012..014 уже выполнили миграцию с `role` как joint: `RoleAnimate`, `RoleMotionPreset`, `RoleTransformable`, `RoleColorToken`, `RoleChartAxis` и другие — всё это реальные файлы в коде. Терминологически вопрос `role` vs `behavior` закрыт **исполнением**, а не решением.

Моя позиция в CLAUDE-009 (`behavior` точнее) остаётся архитектурно обоснованной. Но практически откатывать 653 миграции ради переименования joint нецелесообразно. Это нужно зафиксировать как сделанный выбор в ADR, а не держать открытым: **`role` принят по факту миграции CODER-012..014**. Если владелец хочет `behavior` — решение нужно до следующего пакета миграций.

### 3.2. Определение `slot` — неверное

> "Slot отвечает на вопрос 'Где находится?'"

Это ошибка. `slot` отвечает на вопрос **"Что компонент принимает как содержимое или данные?"** — не "где находится". "Где находится" — это вопрос позиционирования в DOM, что не имеет отношения к интерфейсу. Правильная формулировка: "Какое место для содержимого или данных есть у компонента?"

Эта ошибка важна: из неверного определения будут созданы неверные интерфейсы.

### 3.3. "Static Assembler (Type Builder)"

В CLAUDE-008 было явно зафиксировано: `StructIntersectAll` — это **не Builder, не Factory, не Factory Method**. Это type-level fold / рекурсивная свертка типов. Добавление "(Type Builder)" в скобках воскрешает неверный термин, который мы три часа хоронили в сессии 20260414.

Точное название: **Type-Level Fold** или **Static Type Intersection Fold**. "Builder" — зарезервировано за runtime паттерном GoF.

## IV. По 003-AGENTS-POLICY-REPORT: коллизия `proto` и `role`

### 4.1. Одновременное существование `proto` и `role` — проблема

Таблица joints для кластера `interface` включает оба:

```
proto  — атомарная способность ("чистые поведенческие признаки")
role   — специализированная роль ("расширенные признаки поведения")
```

Это противоречит архитектурному смыслу, который был установлен в сессии. CODER-011 предлагал `proto` объявить **legacy** и **не добавлять** новые сущности в него — вместо этого писать в `behavior` (или `role`). Разделение на "чистые" и "расширенные" поведенческие признаки внутри одного кластера — это возврат к семантической размытости, от которой мы уходили.

Если `role` принят как joint, `proto` должен быть помечен как `[LEGACY]` в таблице, а не стоять наравне с ним.

### 4.2. `provider` в словаре

`interface/provider` внесён в таблицу как легализованный joint. В CLAUDE-008 я фиксировал: этот joint отложен — в проекте 1 файл в `development/`, недостаточно для легализации. Включить его в регламент без ADR преждевременно.

### 4.3. Строка про префикс `I`

> "Префикс `I` (напр. `IClickable`) сохраняется для базовых прототипов."

CODER-011 явно написал: имена `IClickable`, `IFocusable` — **переходные**. Они допустимы до миграции, но не должны быть эталоном для нового кода. Фиксировать их как норму в AGENTS — шаг назад.

## V. По 004-MIGRATION-AND-CODE-REPORT: тревожный прецедент

### 5.1. `CheckboxProps → ContractCheckboxProps` — спорная классификация

В CODER-014 подтверждено: `CheckboxProps`, `RadioProps`, `ToggleProps` → `interface/contract/*-props`. `ButtonProps` в этом списке нет (Gemini перечислил его ошибочно, или он попал в skipped).

Вопрос по существу: правильно ли `CheckboxProps` классифицировать как `contract`?

По правилу сессии (CODER-010): "если сущность является обычной сборкой component props → recipe". `CheckboxProps` — это именно обычная сборка props чекбокса. Логичный таргет: `control/interface/recipe/checkbox → CheckboxRecipe`.

CODER отправил их в `contract`, трактуя как стабильный публичный API элемента управления. Эта логика небесспорна: стабильность API не делает интерфейс системным контрактом — она делает его зрелым recipe.

**Рекомендация:** инспектировать Control Domain — `ContractCheckboxProps`, `ContractRadioProps`, `ContractToggleProps` — и принять решение: оставить в contract (зафиксировав исключение в ADR) или перенести в recipe.

### 5.2. Конфликты: реальная картина

CODER-015 фиксирует: **17 конфликтных целевых файлов** с **72 конфликтными строками** (не 55 файлов, как написал Gemini-004). Основная масса — Calendar domain: разные варианты `CalendarDay`, `CalendarEvent` из разных источников претендуют на один таргет. CODER предложил distinct-таргеты с уточняющими суффиксами — разумно, требует ручного применения.

## VI. По 005-LEGAL-AND-COMMERCIAL-STRATEGY

### 6.1. Что разумно

- Регистрация Программы для ЭВМ в Роспатенте: правильно, это реальный инструмент защиты в РФ.
- Товарный знак SAMO™: правильно, бренд защищается ТЗ, а не патентом.
- Авторское право на тексты ADR, AGENTS: разумно.
- GaaS-модель (платный аудит + IDE-плагин): интересная монетизация, которая монетизирует инструмент, а не методологию.

### 6.2. Что вызывает сомнение

**Метод-патент** (`Способ автоматизированной сборки программных интерфейсов`):

Ст. 1350 ГК РФ исключает из патентования "программы для ЭВМ как таковые". Ключевое слово — "как таковые". Есть прецеденты, когда способ, реализованный через ПО, патентовался — но только при наличии технического результата вне сферы ПО. Рекурсивная свертка типов TypeScript — это чистый software method. Шансы на патент в РФ невысоки.

Альтернатива: патент в США (35 U.S.C. § 101 после Alice Corp. — тоже сложно, но прецеденты есть для "technical improvement" к computer functioning). Если есть бюджет — подача PCT через EP-офис.

### 6.3. Дорожная карта

Q4 2026 — "релиз первой версии платного CLI-аудитора" — реалистично, если к этому времени SAMO-Linter будет покрывать все 36 joints. До этого нужна стабильная версия с тестами.
